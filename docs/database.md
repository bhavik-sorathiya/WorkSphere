# 💾 Database Architecture

WorkSphere utilizes a **Polyglot Persistence** model. This document details the exact schema layouts for both PostgreSQL and MongoDB.

## 1. PostgreSQL (Relational Core)

PostgreSQL is strictly reserved for data that requires absolute ACID compliance, relational integrity, and strict boundaries.

### 1.1 The Organization Boundary
Everything cascades down from the `Organization`.
*   `Organization`: The top-level multi-tenant boundary.
*   `WorkspaceSetting`: 1-to-1 mapping with `Organization`. Dictates rules like `allowMultipleWorkspaces`.
*   `OrganizationInvite`: Tracks pending invites (bound by email and `OrganizationId`).
*   `WorkspaceJoinRequest`: Tracks users asking for permission to join across strict workspace boundaries.

### 1.2 The Project Boundary
Projects exist strictly within an `Organization`.
*   `Project`: The core collaborative unit. Enforces `@@unique([organizationId, slug])`.
*   `ProjectMember`: Junction table mapping a `User` to a `Project` with a specific `Role`.
*   `Channel`: A chat channel belonging to a `Project`.

### 1.3 The Task & Entity Boundary
Tasks belong to a `Project`.
*   `Task`: Has states (TODO, IN_PROGRESS, DONE) and priorities.
*   `TaskAssignee`: Maps multiple `User`s to a `Task`.
*   `TaskComment`: Maps a `User`'s comment to a `Task`.

### 1.4 The User (Identity)
*   `User`: The primary key `id` is a 1:1 map with the Clerk `user_id`.
*   `UserOrganizationRole`: Junction table mapping a `User` to an `Organization` with a specific macro `Role`.

> [!WARNING]
> Almost every relation in `schema.prisma` utilizes `@relation(..., onDelete: Cascade)`. Do not remove these directives. They are critical to preventing orphaned rows when users delete Projects or Organizations.

## 2. MongoDB (High-Velocity Data)

MongoDB is reserved for unstructured, fast-moving data that does not require relational integrity checks.

### 2.1 ActivityLog
*   **Purpose:** The central audit trail.
*   **Schema:** `{ action: String, userId: String, projectId: String, metadata: Mixed, createdAt: Date }`
*   **Usage:** Fire-and-forget inserts that do not block the main PostgreSQL transaction.

### 2.2 Message
*   **Purpose:** Real-time chat history.
*   **Schema:** `{ channelId: String, senderId: String, content: String, attachments: Array, isSystem: Boolean, timestamp: Date }`
*   **Usage:** Queried via `$lt` cursors for infinite scrolling in the UI.

### 2.3 Document
*   **Purpose:** Collaborative BlockNote text documents.
*   **Schema:** `{ projectId: String, title: String, content: Mixed, createdBy: String, updatedBy: String, createdAt: Date, updatedAt: Date }`
*   **Usage:** `content` is a highly nested, unstructured JSON array generated by the BlockNote editor, which is fundamentally incompatible with strict SQL schemas.
